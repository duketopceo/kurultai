What we are ideating on

Phase 1 of Kurultai: the thinnest vertical slice that lets a developer index local knowledge and search it from CLI and MCP — without distillation, enterprise multi-tenancy, or plugin runtimes. The codebase is still a trait stub (~574 LOC); issue #27 is the work-order source of truth.

Grounding context

  • Codebase: Rust 2021 CLI + lib; traits for Connector, Store, Embedder, QueryEngine. main.rs never imports the library. No SQL, no tests, no CI, no MCP module. rusqlite present; sqlite-vec not yet a dependency. Embedder stub returns all-zero vectors.
  • Plan: GitHub #27 Phase 1 — framework (done), sqlite-vec, OpenRouter embeddings, Obsidian/AppFlowy, CLI wire-up, MCP+installer, CI gates. Audience: developer.
  • External: Canonical hybrid schema (base + FTS5 + vec0 shared rowid, RRF k≈60). Prior art: obsidian-hybrid-search, opencode-memory, Loci, markdown-vault-mcp, codebase-memory-mcp installers. Convergent Phase 1: MCP stdio, 4–8 read tools, hash-incremental index, defer LLM distillation.
  • Past learnings: none (docs/solutions/ absent).

1. FTS-First Boot + Zero-Vector Guard

Embeddings & providers Conf 92% Cx Medium

Description: Index always fills base table + FTS5. Vector embed runs only when a real provider key exists. search/MCP default to FTS until vectors exist, then hybrid RRF. Reject all-zero (or near-zero-norm) vectors at upsert so the stub embedder cannot silently poison the store.

Basis: direct: OpenRouterEmbedder::embed returns vec![0.0; dim] (src/embed/mod.rs). external: obsidian-hybrid-search / markdown-vault-mcp ship lexical before vectors; Alex Garcia hybrid treats FTS and vec as parallel arms.

Rationale: Delivers useful retrieval on day one without API spend, and fails loud when embeddings are fake — the current worst failure mode.

Downsides: Two retrieval modes to document; users may think “vector search is broken” when it is intentionally offline.

2. Schema-at-Open + Embedding Contract

Storage & schema Conf 90% Cx Medium

Description: On SqliteVecStore::open, run one idempotent DDL block: atoms + FTS5 + vec0 sharing rowid, dimension from config. Persist store_meta (embed_model, embed_dim, schema version). Refuse mixed dimensions; model change requires explicit --rebuild-vectors. No migrations framework in Phase 1.

Basis: direct: no SQL/migrations in repo; Cargo.toml has rusqlite only; README/types.rs fix 3072-dim. external: sqlite-vec #48 hybrid pattern; dim is schema-fixed across prior art.

Rationale: Unblocks upsert + hybrid search in one atomic step and prevents silent vector corruption when providers change.

Downsides: Dimension changes are loud and costly (full re-embed); acceptable for Phase 1.

3. Raw-First Atoms + Filesystem Connector

Connectors & indexing Conf 88% Cx Low–Medium

Description: Connectors emit title + raw content + path metadata only; leave summary/question/resolution empty. Ship a filesystem .md connector using existing glob/notify deps; treat Obsidian as vault_path → filesystem root for Phase 1. Defer AppFlowy API and LLM distillation.

Basis: direct: Obsidian/AppFlowy return []; no distillation module; notify/glob unused. external: successful Phase 1 tools index raw chunks first; distillation is a precision lever later.

Rationale: Shortest path from stub to atoms-in-store; validates the whole loop without cloud note APIs.

Downsides: AppFlowy (#4) slips; chunking quality is naive until distillation returns.

4. Shared Runtime Kernel

CLI, config & wiring Conf 91% Cx Medium

Description: Extract kurultai::app (load/init config → store + embedder + connector registry → orchestrate). main.rs becomes a thin clap wrapper. kurultai init writes the Rust Config shape (single source of truth) so README/TOML drift ends. Honest registry: only implemented kinds report ready.

Basis: direct: main.rs never use kurultai::; status hardcodes five sources; README TOML shape ≠ Config struct.

Rationale: One kernel means every store/search improvement compounds to CLI and MCP without divergent paths; makes integration tests call the library directly.

Downsides: Requires a short CLI refactor before features land; resist the urge to rewrite clap surface.

5. MCP Stdio Primary + install/doctor

MCP, installer & CI gates Conf 87% Cx Medium–High

Description: Ship kurultai mcp (stdio) with 4 read tools: search, read_atom, status, reindex — sharing the runtime kernel. CLI keeps init/index/status (and thin search for humans). kurultai install writes Cursor/Claude Code MCP config; doctor checks binary path, store, embedder health.

Basis: direct: no src/mcp/, README promises MCP. external: Loci/opencode-memory MCP+shared SQLite; codebase-memory-mcp install pattern.

Rationale: Without MCP, Kurultai is invisible to the agents that are the product’s distribution channel.

Downsides: MCP crate choice and host quirks; keep tool surface capped at 4–8.

6. Golden-Path CI (fixture vault as Phase 1 spec)

MCP, installer & CI gates Conf 89% Cx Low

Description: One integration test is the Phase 1 definition of done: fixture vault (3–5 markdown files) → index → assert FTS hit (and vector hit when key present). Add GitHub Actions cargo test + clippy. Optionally MCP tools/list smoke after MCP lands.

Basis: direct: no tests/, no .github/, no #[test]. Aligns with #23 testing gates.

Rationale: Turns “Phase 1 done” from a narrative into a failing/passing gate every PR can enforce.

Downsides: Needs sqlite-vec in CI (bundled extension or skip vector arm when unavailable).

7. Content-Hash Incremental + index --watch

Connectors & indexing Conf 78% Cx Medium

Description: Drive upserts from content-addressed atom ids; skip re-embed when hash unchanged. Replace the sleeping daemon stub with kurultai index --watch (debounced notify). Keep --full as rebuild, not a second connector API forever.

Basis: direct: KnowledgeAtom.id documented as content hash; daemon is a sleep loop; notify unused. external: hash-incremental is table stakes. Verifier: weak on “hash already implemented” — it is documented intent, not code yet.

Rationale: Makes re-index cheap and removes a misleading always-on HTTP fiction from Phase 1.

Downsides: Orphan deletion / manifest sync still needed; watch mode is optional sugar after batch index works.

Rejection summary

Pond-second before AppFlowy

Why reject: Verifier weak; conflicts with #27 order (Obsidian/AppFlowy before Pond). Strong as Phase 4 differentiator, not Phase 1.

Init-only / drop TOML loader as sole answer

Why reject: Subsumed by Shared Runtime Kernel (init writes Config shape). Prefer one wiring story.

Remove CLI search entirely

Why reject: Too aggressive; keep thin CLI search for humans while MCP is primary for agents.

Named embedding spaces / multi-root indexes / batch-API-only embed

Why reject: Over-scope for Phase 1; fingerprint + rebuild covers the real risk.

Analogy-only metaphors (cadastral, emulsion, harbor, customs)

Why reject: Useful framing, not distinct product moves beyond survivors above.
